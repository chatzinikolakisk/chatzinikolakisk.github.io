The Test Pyramid
& Test-Driven Development

Building confidence through testing strategy

Welcome to session 3. Today we'll cover two related topics: the test pyramid as a framework for WHERE to invest in testing, and TDD as a practice for HOW to write tests. Both are essential tools in a tech lead's toolkit.
Part 1

The Test Pyramid

A cost model for testing investment

The test pyramid isn't just a diagram — it's a decision-making tool. As a tech lead, you need to think about how your team allocates testing effort.

Three Layers

Three layers, each with different tradeoffs. Unit tests at the base: fast, cheap, isolated — test a single function or class. Integration in the middle: test how components work together, like your service talking to a database. E2E at the top: test the whole system end-to-end, like a real user flow. The pyramid shape tells you WHERE to invest most.

What lives where?

Unit

  • Calculate discount price
  • Validate email format
  • Parse API response
  • Business logic rules

Integration

  • API endpoint returns correct data
  • DB query + service layer
  • Auth middleware chain
  • External API contracts

E2E

  • User signup → email → login
  • Purchase flow end-to-end
  • Critical happy paths
  • Cross-service workflows
Let's make this concrete. Unit tests cover isolated logic — a function that calculates a price, validates an input. Integration tests verify components working together — does your endpoint actually talk to the database correctly? E2E tests cover real user journeys — can someone actually sign up and use the product? Each layer answers a different question.

It's a cost model

Every test you write has a maintenance cost:

  • Unit: milliseconds to run, easy to fix
  • Integration: seconds to run, moderate setup
  • E2E: minutes to run, brittle, hard to debug

The pyramid tells you: invest most where the cost is lowest

Typical execution time

~5ms

per unit test

~500ms

per integration test

~30s+

per E2E test

The pyramid isn't about dogma — it's about economics. Unit tests cost almost nothing to run and are easy to maintain. E2E tests are expensive: slow to run, brittle, hard to debug when they fail. So you want the bulk of your coverage at the unit level. This doesn't mean E2E isn't valuable — it absolutely is for critical paths. But it shouldn't be your primary testing strategy.

The Anti-Pattern: Ice Cream Cone

✓ Pyramid (healthy)

✗ Ice Cream Cone

Slow CI · flaky tests nobody trusts · expensive to maintain

This is the most common anti-pattern you'll see. Teams pile up E2E and manual tests because they "feel" safer, and under-invest in unit tests. The result: a slow CI pipeline, tests that flake randomly, and a test suite the team stops trusting. As a tech lead, recognizing this shape and correcting it is one of the highest-leverage things you can do.

The Tech Lead Question

"Is our testing investment distributed well?"

Not just "do we have tests?" but:

  • Are we fast at the base? Can devs get feedback in seconds?
  • Are we selective at the top? Only critical paths in E2E?
  • Do developers trust the test suite?

The pyramid is the shape — your job is to keep it that way.

The shift from "senior engineer" to "tech lead" thinking is this: you stop asking "do we have tests?" and start asking "is our test investment making us faster or slower?" A healthy pyramid means fast feedback loops, confident deployments, and a team that trusts the green build. An inverted pyramid means slow CI, flaky tests, and developers who skip running tests locally because "they always break anyway."

The pyramid tells you where to test

Now let's talk about how

So the pyramid gives us the WHERE — invest heavily at the unit level, be selective at the top. Now let's talk about HOW to actually write those tests effectively. Enter TDD.
Part 2

Test-Driven Development

A design practice disguised as a testing practice

TDD is often misunderstood as "just writing tests first." It's actually a design discipline — it changes how you think about code before you write it.

The Cycle: Red → Green → Refactor

1. RED
Write a failing test
→
2. GREEN
Make it pass (minimal)
→
3. REFACTOR
Clean up the code

repeat in small steps

The cycle is simple. Step 1: write a test that describes what you want — it fails because the code doesn't exist yet. Step 2: write the minimum code to make it pass. Don't over-engineer, just pass the test. Step 3: now that it's working, clean it up — remove duplication, improve naming, restructure. Then repeat. The key is small steps — each cycle should be minutes, not hours.

A Quick Example

① RED — Write the test first

public function test_apply_discount(): void
{
    $price = calculatePrice(
        base: 100,
        discountPct: 20
    );
    $this->assertSame(80.0, $price);
}

calculatePrice doesn't exist yet → test fails ✗

② GREEN — Simplest thing that works

function calculatePrice(float $base, float $discountPct): float
{
    return $base * (1 - $discountPct / 100);
}

Test passes ✓ → now refactor if needed

Notice: the test defined the interface before calculatePrice existed.

Simple example. We write the test first — we decide what the function should be called, what arguments it takes, and what it should return. The function doesn't exist yet, so the test fails. Then we write the simplest code to make it pass. The key observation: by writing the test first, we designed the interface before we wrote the code. That's the real power of TDD.

TDD is a Design Tool

Writing tests first forces you to:

  • Think about interfaces before implementation
  • Keep code modular — hard-to-test code is often hard-to-use code
  • Produce loosely coupled components

THE INSIGHT

If your code is hard to test,
it's probably hard to maintain.

TDD gives you that signal before you've built too much.

This is the most important slide about TDD. Most people think TDD is about catching bugs. It's really about design feedback. When you write the test first, you're the first "user" of your own code. If it's awkward to set up, if it needs too many dependencies, if the interface is confusing — you find out immediately. Hard-to-test code is a design smell, and TDD surfaces that smell early, before you've built layers on top of it.

Why This Matters for Tech Leads

Confidence

Tests as a safety net: your team can change code without fear. New contributors can modify existing code on day one.

Sustainability

Modular, well-tested code stays maintainable as the team and codebase grow. Less "don't touch that" code.

Velocity

Counterintuitively, TDD often speeds teams up. Less debugging, fewer regressions, faster onboarding.

As a tech lead, you're responsible for a codebase that multiple people touch. TDD gives you three things. Confidence: developers can change code without fear because tests have their back. Sustainability: the codebase stays modular and maintainable as it grows. Velocity: yes, writing tests takes time upfront, but you save it many times over in reduced debugging, fewer regressions, and faster onboarding of new team members.

Being Pragmatic

TDD shines when...

  • Business logic & domain rules
  • Clear inputs → expected outputs
  • APIs and service interfaces
  • Bug fixes (write test that reproduces, then fix)
  • Refactoring existing code

TDD struggles when...

  • Exploratory / prototyping work
  • UI layout and visual design
  • You don't yet know what to build
  • Heavily integrated external systems
  • One-off scripts & spikes

A good tech lead promotes TDD as a default mindset, not a rigid mandate.

Be honest with your team: TDD isn't always practical. It's great for business logic, well-defined interfaces, bug fixes (write a test that reproduces the bug, then fix it). It's harder when you're exploring, prototyping, or doing UI work where the output is visual. The mature approach: TDD is the default for logic and interfaces, but use your judgment. A tech lead who mandates 100% TDD everywhere will create friction; one who never promotes it misses its benefits.

Addressing the Pushback

❝ "It slows me down"

It slows you down now. It speeds up the team next month. Tech leads think in weeks, not hours.

❝ "I'll write the tests after"

After-the-fact tests verify implementation, not behavior. You also lose the design benefit.

❝ "Our code is too coupled for TDD"

That's exactly why you need it. TDD makes coupling visible and painful — which motivates decoupling.

You'll hear these objections from your team. "It slows me down" — yes, individually and in the short term. But as a tech lead you think about team velocity over time, not individual speed today. "I'll write tests after" — tests written after the fact tend to test implementation details rather than behavior, and you've lost the design feedback. "Our code is too coupled" — that's the strongest argument FOR TDD, not against it. The pain of testing coupled code is the signal that drives decoupling.
