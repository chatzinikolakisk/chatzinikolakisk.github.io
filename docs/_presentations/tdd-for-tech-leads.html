Tech Lead School · Session 3

The Test Pyramid
& Writing Unit-Testable Code

Shifting from "we have tests" to "our tests make us better engineers"

Welcome to session 3. You already know what tests are. You're already writing them. Today isn't about convincing you to test — it's about shifting how you think about what a test actually is, and what writing the right kind of test forces you to do to your code.
Part 1

The Test Pyramid

Most teams have it upside down — and don't know it

The test pyramid is a decision-making tool, but more importantly it's a mirror. Most teams look at it and think they're doing fine. They're not. They're mostly writing integration tests and calling them unit tests. Let's look at why that matters.

Three Layers

Three layers. The pyramid shape tells you WHERE to invest most — the base. But most teams flip this instinctively. They write integration tests because they feel more real, more complete. And unit tests feel trivial. That instinct is wrong, and we'll get into exactly why.

What lives where?

Unit

  • Calculate discount price
  • Validate email format
  • Parse API response
  • Business logic rules

Integration

  • API endpoint returns correct data
  • DB query + service layer
  • Auth middleware chain
  • External API contracts

E2E

  • User signup → email → login
  • Purchase flow end-to-end
  • Critical happy paths
  • Cross-service workflows
Let's make this concrete. Unit tests cover isolated logic — a function that calculates a price, validates an input. Integration tests verify components working together — does your endpoint actually talk to the database correctly? E2E tests cover real user journeys — can someone actually sign up and use the product? Each layer answers a different question.

Cost is only half the story

Every test has a maintenance cost:

  • Unit: milliseconds to run, easy to fix
  • Integration: seconds to run, moderate setup
  • E2E: minutes to run, brittle, hard to debug

But the bigger point: an integration test can pass while your design is still bad. A unit test can't.

Typical execution time

~5ms

per unit test

~500ms

per integration test

~30s+

per E2E test

Cost matters, but it's not the most important thing. Here's what is: integration tests give you false confidence about your design. They tell you "these pieces work together" but not "this code is well-structured." A unit test is brutal — if your code is badly designed, you'll feel it immediately when you try to write one.

The Anti-Pattern: Ice Cream Cone

✓ Pyramid (healthy)

✗ Ice Cream Cone

This is probably your team right now — and it's not your fault

This is the most common pattern I've seen. Teams pile up integration and E2E tests because they feel more real — you're testing something you can see. Unit tests feel almost too small. But the ice cream cone means slow CI, flaky tests, and a suite nobody trusts. The root cause isn't laziness — it's that nobody taught them what a real unit test is.

The Questions to Ask Your Team

"What does a unit test actually mean to us?"
  • Do your developers know the difference between a unit and an integration test?
  • If their definition is fuzzy, their TDD will be wrong too
  • Can devs run the full suite locally in under 30 seconds?
  • Does a green build mean good code, or just working plumbing?

Before fixing the pyramid, fix the understanding.

The most important shift I made as a tech lead wasn't introducing TDD — it was getting alignment on what a unit test actually is. If your team conflates unit and integration tests, TDD becomes just "write a test that spins up the database before you write the feature." That's not TDD. It's just slower integration testing.

The pyramid tells you where to invest

So why can't most teams get there?

The pyramid makes sense to everyone when they see it. But most teams still end up with the cone. Why? Because building a proper unit test base requires a different way of thinking about code. Let's talk about that.
Part 2

The Mentality Shift

From "we test what we built" to "we build so it can be tested"

The real shift isn't about process — it's about design. The question isn't "do we write tests before or after?" It's "do we write code that CAN be unit tested?" That's a fundamentally different kind of code.

What isolation really means

Integration test — tells you the plumbing works

public function test_user_is_created(): void
{
    $user = $this->userService
        ->create('alice@example.com');

    $this->assertDatabaseHas('users', [
        'email' => 'alice@example.com'
    ]);
}

Needs a real DB · slow · tells you nothing about design

Unit test — tells you the logic is correct

public function test_discount_is_applied(): void
{
    $price = calculatePrice(
        base: 100,
        discountPct: 20
    );

    $this->assertSame(80.0, $price);
}

No DB · no HTTP · no filesystem · pure logic

A unit test has zero external dependencies. No database, no network, no filesystem, no other services. If you need to mock a database to run your test, that's an integration test — and that's fine, but call it what it is. The rule of thumb: if your test can't run offline in a fresh PHP process with no container, it's not a unit test.

What unit-testable code looks like

① Write the test — define the interface

public function test_apply_discount(): void
{
    $price = calculatePrice(
        base: 100,
        discountPct: 20
    );
    $this->assertSame(80.0, $price);
}

calculatePrice doesn't exist yet → test fails ✗

② Write the implementation — minimal, focused

function calculatePrice(float $base, float $discountPct): float
{
    return $base * (1 - $discountPct / 100);
}

Test passes ✓ → now refactor if needed

Notice what this function is not doing: no DB call, no HTTP, no global state. Clear input → clear output. That's it.

This function is trivially unit-testable because it does exactly one thing. It takes two numbers and returns one number. No setup, no mocking, no containers. If your functions looked like this, your test suite would be fast, reliable, and trustworthy. The question is: how do you get your codebase there?

The question that changed everything

"How would I unit test this?"

We didn't always know how to write good code. But whenever we got stuck on a design decision, we asked this one question — and it guided us every time.

  • If the answer was "I'd need to spin up a DB" → the class was doing too much
  • If the answer was "I'd need to mock 5 things" → the dependencies were wrong
  • If the answer was "I'd need to change global state" → that's a design smell

The pain of testing is the signal. Don't suppress it — follow it.

This is the most personal thing I can share with you. We were building a system and we weren't sure how to structure something. One of the engineers said "how would we even unit test this?" and we realised we couldn't. That was the signal. We refactored until the answer was simple. That discipline — following the pain of testing back to a design problem — is something I still use every day.

Single Responsibility: the north star

A function or class that does one thing:

  • Is trivially unit-testable
  • Has clear inputs and outputs
  • Can be changed without fear
  • Can be understood in isolation

SRP isn't just a clean code principle — it's the prerequisite for unit testability.

THE RULE

If your test setup is painful, your code is doing too much.

Stop trying to write the test. Start asking what needs to be split.

Single Responsibility is often taught as a style preference. It's not. It's a forcing function. When you commit to writing unit-testable code, SRP stops being optional — it becomes mandatory, because code that does too many things cannot be tested in isolation. And code that can't be tested in isolation will rot.

Why This Matters for Tech Leads

Confidence

Your team can change code without fear. New engineers can modify existing code on day one.

Sustainability

Modular, well-tested code stays maintainable as the codebase grows. Less "don't touch that" code.

Velocity

Less debugging, fewer regressions, faster onboarding. The upfront cost pays back many times over.

AI-Readability

Unit-testable code — clear inputs, single responsibility, no hidden state — is exactly what AI agents parse and reason about best. This is how we write code for the future.

I want to highlight the fourth one because it's where we're heading. AI coding assistants are already part of our workflow. The code they work best with — and the code they generate that's easiest to review and trust — has clear boundaries, single responsibility, no hidden dependencies. That's not a coincidence. It's the same thing unit testability demands. Writing unit-testable code today means writing AI-compatible code tomorrow.

Being Pragmatic

TDD shines when...

  • Business logic & domain rules
  • Clear inputs → expected outputs
  • APIs and service interfaces
  • Bug fixes (write test that reproduces, then fix)
  • Refactoring existing code

TDD struggles when...

  • Exploratory / prototyping work
  • UI layout and visual design
  • You don't yet know what to build
  • Heavily integrated external systems
  • One-off scripts & spikes

A good tech lead promotes TDD as a default mindset, not a rigid mandate.

Be honest with your team: TDD isn't always practical. It's great for business logic, well-defined interfaces, bug fixes (write a test that reproduces the bug, then fix it). It's harder when you're exploring, prototyping, or doing UI work where the output is visual. The mature approach: TDD is the default for logic and interfaces, but use your judgment. A tech lead who mandates 100% TDD everywhere will create friction; one who never promotes it misses its benefits.

Addressing the Pushback

❝ "It slows me down"

It slows you down now. It speeds up the team next month. Tech leads think in weeks, not hours.

❝ "I'll write the tests after"

After-the-fact tests verify implementation, not behavior. You also lose the design benefit.

❝ "Our code is too coupled for TDD"

That's exactly why you need it. TDD makes coupling visible and painful — which motivates decoupling.

You'll hear these objections from your team. "It slows me down" — yes, individually and in the short term. But as a tech lead you think about team velocity over time, not individual speed today. "I'll write tests after" — tests written after the fact tend to test implementation details rather than behavior, and you've lost the design feedback. "Our code is too coupled" — that's the strongest argument FOR TDD, not against it. The pain of testing coupled code is the signal that drives decoupling.

Putting It Together

Pyramid

Where to invest

+

Unit-testable code

SRP · isolation · clear interfaces

=

Clean, maintainable,
AI-ready code

Ship without fear. Grow without rot.

Ask not "do we have tests?" Ask "can we unit test this — and if not, why not?"
That one question — "can I unit test this?" — is the compass. It leads to SRP, to loose coupling, to clean interfaces. And clean interfaces are what lets your team move fast, what lets new engineers contribute on day one, and what lets AI tools work with your codebase effectively. This is the direction we're moving. Start with the question.
